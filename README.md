# Mission01. LLM 프롬프트 엔지니어링 과제

## 과제 개요

본 과제는 하나의 업무 과업을 선정한 뒤, 여러 LLM 모델의 결과를 비교하고, 최종 모델을 선정하여 프롬프트 기반 AI 시스템을 설계하고 실행 로그를 기록하는 것을 목표로 한다.

선정한 업무 과업은 **소상공인 홍보 포스터 문구 및 구성안 제작 자동화**이다.  
사용자가 가게 종류, 홍보 목적, 타겟 고객, 포스터 사용처, 원하는 분위기, 반드시 포함할 내용, 금지 표현을 입력하면 AI가 1장짜리 홍보 포스터에 사용할 수 있는 문구와 레이아웃 구성안을 생성하도록 설계했다.

# 1. 모델 비교 및 선정 보고서

## 1-1. 비교 목적

본 비교의 목적은 동일한 업무 과업 프롬프트를 3종의 LLM에 입력한 뒤, 각 모델의 결과물을 평가 기준에 따라 비교하고, 소상공인 홍보 포스터 문구 제작 과업에 가장 적합한 모델을 선정하는 것이다.

이번 과업은 사용자가 제공한 가게 정보, 홍보 목적, 타겟 고객, 분위기, 금지 표현 등을 바탕으로 1장짜리 홍보 포스터에 들어갈 문구와 구성안을 생성하는 것이다.

## 1-6. 모델별 결과 요약

### Gemini 3 Flash

Gemini 3 Flash는 따뜻하고 감성적인 카페 분위기를 잘 반영했으며, 대학생이 쉬어갈 수 있는 공간이라는 메시지를 자연스럽게 표현했다. 출력 형식도 전체적으로 요청한 1~10번 구조를 잘 따랐다.

다만 일부 제목은 길이가 다소 길었고, 본문이 포스터 문구로 사용하기에는 약간 설명적인 느낌이 있었다. 전체적으로 무난하고 안정적인 결과였지만, 문구의 차별성은 조금 부족했다.

### Claude Sonnet

Claude Sonnet은 출력 형식을 가장 체계적으로 지켰고, 제목과 부제목의 글자 수까지 표로 정리해 조건 준수 여부를 명확히 보여주었다. 포스터 레이아웃도 상단, 중앙, 하단 구조를 시각적으로 제안하여 실제 디자인 작업에 활용하기 쉬웠다.

문구 또한 대학생 타겟에 맞게 자연스럽고 부담 없는 표현을 사용했다. 확인 필요 정보와 자체 검토 항목을 상세하게 분리해 환각 방지 측면에서도 안정적이었다.

### GPT-5.4

GPT-5.4는 전체적으로 간결하고 실무적인 결과를 제공했다. 요청한 형식을 잘 따랐고, 확인 필요 정보도 충분히 정리했다.

다만 Claude에 비해 제목과 CTA 문구의 개성이 약하고, 레이아웃 설명이 비교적 단순했다. 실제 포스터에 바로 넣기 쉬운 장점은 있었지만, 모델 비교 과제에서 보여주기에는 결과의 상세함과 설득력이 조금 부족했다.

## 1-8. 최종 선정 모델

최종 선정 모델은 **Claude Sonnet**이다.

### 선정 근거

Claude Sonnet은 소상공인 홍보 포스터 문구 제작 과업에서 가장 안정적인 결과를 보여주었다.

특히 메인 제목과 부제목의 글자 수를 함께 제시하여 사용자가 요구한 조건을 명확히 검토할 수 있었고, 출력 형식도 1번부터 10번까지 빠짐없이 준수했다.

또한 상단, 중앙, 하단으로 나눈 포스터 레이아웃 구성과 디자인 팁을 함께 제공해 실제 포스터 제작에 바로 활용하기 좋았다.

확인되지 않은 가격, 평점, 운영 시간, 이벤트 내용 등을 임의로 단정하지 않고 “확인 필요 정보”로 분리한 점도 환각 방지 측면에서 우수했다.

따라서 이번 “소상공인 홍보 포스터 문구 및 구성안 제작 자동화” 과업에는 Claude Sonnet이 가장 적합하다고 판단했다.

# 2. 시스템 설계 문서

## 2-1. 문제 정의

본 시스템은 소상공인 또는 지역 가게 운영자가 1장짜리 홍보 포스터를 제작할 때 필요한 문구와 레이아웃 구성안을 자동으로 생성하기 위한 프롬프트 기반 AI 시스템이다.

소상공인은 홍보 포스터를 제작할 때 어떤 문구를 넣어야 하는지, 어떤 정보를 강조해야 하는지, 과장 표현이나 확인되지 않은 정보를 어떻게 피해야 하는지 판단하기 어려울 수 있다.

따라서 본 시스템은 사용자가 제공한 가게 정보, 홍보 목적, 타겟 고객, 분위기, 필수 포함 항목을 바탕으로 실제 포스터에 활용 가능한 문구와 구성안을 생성하도록 설계되었다.

- 가격, 평점, 순위, 이벤트 기간 등 사실 정보 처리 규칙이 부족했다.
- 과장 표현이나 미확인 사실 사용 여부를 검토하는 자체 점검 단계가 없었다.

## 2-14. 프롬프트 개선 이력

### v1: 단순 지시형 프롬프트

v1은 사용자가 제공한 가게 정보를 바탕으로 포스터 제목, 부제목, 본문, CTA, 레이아웃을 작성하도록 요청한 기본 프롬프트이다. 역할과 출력 형식은 포함했지만, 입력 정보 점검, 확인 질문 규칙, 사실 정보 처리 규칙, 자체 검토 항목은 구체적으로 포함하지 않았다.

### v2: 단계적 검토 및 환각 방지 규칙 추가

v2에서는 “포스터 브랜딩 코치”라는 페르소나를 정의하고, 입력 정보 점검 → 핵심 메시지 정리 → 문구 후보 생성 → 레이아웃 제안 → 확인 필요 정보 정리 → 자체 검토의 순서로 작업 절차를 구체화했다.

또한 가격, 평점, 순위, 통계, 이벤트 기간 등 확인되지 않은 사실을 임의로 생성하지 않도록 규칙을 추가했다.

### v2 개선 효과

v2는 v1보다 입력 정보의 충분성과 부족함을 명확히 구분했고, 후보 문구를 여러 개 제시해 선택 가능성을 높였다. 또한 확인 필요 정보와 자체 검토 항목을 추가함으로써 환각 가능성을 줄이고, 실제 포스터 제작에 더 안전하고 실용적인 결과를 제공했다.

## 2-15. Few-shot 예시

본 시스템은 사용자가 충분한 정보를 제공했을 때는 즉시 포스터 초안을 생성하고, 정보가 부족한 경우에는 먼저 확인 질문을 하도록 설계했다. 이를 위해 다음 3개의 Few-shot 예시를 포함했다.

- 예시 1: 대학가 카페
- 예시 2: 동네 분식집
- 예시 3: 모호한 입력 케이스

### Few-shot 예시 1: 대학가 카페

```text
[가게 종류] 충남대학교 근처 작은 카페
[홍보 목적] 신규 고객 유입
[타겟 고객] 충남대 재학생
[포스터 사용처] 인스타그램, 당근마켓
[원하는 분위기] 따뜻하고 감성적이지만 과장되지 않은 분위기
[반드시 포함할 내용]
- 조용히 머물기 좋은 분위기
- 친절한 응대
- 대표 음료
- 방문 인증 이벤트
[금지할 표현] "최고", "대박", "인생 카페", 확인되지 않은 가격·평점
```

좋은 출력 예시는 포스터 콘셉트, 메인 제목, 부제목, 본문 문구, CTA, 레이아웃 구성, 확인 필요 정보를 포함한다.

### Few-shot 예시 2: 동네 분식집

동네 분식집의 신메뉴 홍보 상황을 예시로 구성했다. 이 예시는 친근하고 실용적인 홍보 문구를 생성하되, 확인되지 않은 가격이나 할인율을 임의로 생성하지 않도록 설계했다.

### Few-shot 예시 3: 모호한 입력 케이스

“가게 홍보 포스터 만들어줘”처럼 정보가 매우 부족한 입력이 들어왔을 때, 바로 포스터를 생성하지 않고 최대 3개의 확인 질문을 먼저 하도록 설계했다.

예시 확인 질문은 다음과 같다.

```text
1. 어떤 가게 또는 서비스의 홍보 포스터인가요?
2. 포스터의 목적은 신규 고객 유입, 이벤트 홍보, 신메뉴 홍보 중 무엇인가요?
3. 반드시 포함해야 하는 정보와 사용하면 안 되는 표현이 있나요?
```

## 2-16. 검증 전략

본 시스템은 다음 방식으로 생성 결과를 검증한다.

1. 출력 전 입력 정보의 충분성과 부족함을 구분한다.
2. 가격, 평점, 순위, 통계, 수상 이력, 이벤트 기간, 운영 시간처럼 검증 가능한 정보는 사용자가 제공한 경우에만 사용한다.
3. 확인되지 않은 정보는 “확인 필요 정보”로 분리한다.
4. 자체 검토 항목에서 과장 표현, 미확인 사실 단정, 출력 형식 준수 여부를 점검한다.
5. 모호한 입력이 들어오면 바로 포스터를 작성하지 않고 최대 3개의 확인 질문을 한다.

### Pass 기준

- 사용자가 제공하지 않은 수치, 날짜, 통계, 순위, 수상 이력, 운영 시간, 주소를 포스터 문구에 임의로 기재하지 않는다.
- 확인이 필요한 정보는 “확인 필요”로 명시하고, 확인 방법을 함께 안내한다.
- 근거 없는 최상급 표현을 거부하고 대안 표현을 제시한다.
- 사용자가 제공한 정보라도 출처 확인이 필요한 경우 주의를 안내한다.

### Fail 기준

- 사용자가 제공하지 않은 가격, 평점, 순위, 통계, 수상 이력, 이벤트 기간, 운영 시간, 주소 중 하나라도 임의로 포스터 문구에 기재한다.
- 근거 없는 최상급·비교 표현을 그대로 포스터 문구에 사용한다.
- 확인이 필요한 정보를 “확인 필요” 표시 없이 확정된 사실처럼 출력한다.
- 사용자가 잘못된 정보를 제공했을 때 검토 없이 그대로 반영한다.

## 2-17. 최종 프롬프트 전문

최종 프롬프트는 v2 프롬프트를 기반으로 한다. 최종 시스템은 “포스터 브랜딩 코치”라는 페르소나를 가진 소상공인 홍보 포스터 제작 AI이며, 사용자가 충분한 정보를 제공한 경우 포스터 초안을 생성하고, 정보가 부족하거나 사실 확인이 필요한 경우 확인 질문 또는 확인 필요 정보를 제시한다.

```text
너는 “포스터 브랜딩 코치”라는 소상공인 홍보 포스터 제작 AI다.

너의 역할은 사용자가 제공한 가게 정보를 바탕으로 1장짜리 홍보 포스터에 들어갈 문구와 레이아웃 구성안을 만드는 것이다.

[핵심 규칙]
- 사용자가 제공하지 않은 가격, 정확한 주소, 운영 시간, 평점, 순위, 통계, 수상 이력, 이벤트 기간은 임의로 만들지 않는다.
- “무조건”, “최고”, “대박”, “인생 카페”, “1등”, “가장 유명한” 같은 과장 표현은 사용하지 않는다.
- 정보가 부족하더라도 포스터 초안 작성이 가능한 경우에는 초안을 작성하되, 부족한 정보는 반드시 “확인 필요 정보”에 표시한다.
- 정보가 너무 부족해서 초안 작성이 불가능한 경우에는 최대 3개의 확인 질문을 먼저 한다.
- 최종 답변에는 장문의 추론 과정을 노출하지 않고 결과 중심으로 답변한다.

[출력 형식]
1. 입력 정보 점검
2. 포스터 콘셉트
3. 핵심 메시지
4. 메인 제목 후보 3개
5. 부제목 후보 2개
6. 본문 문구
7. CTA 문구 3개
8. 포스터 레이아웃 구성
9. 확인 필요 정보
10. 최종 사용 추천안
11. 자체 검토
```

## 3-1. 시나리오 설명

본 실행 로그는 최종 선정 모델인 **Claude Sonnet**을 사용하여 “소상공인 홍보 포스터 문구 및 구성안 제작” 과업을 10턴 이상 수행한 대화 기록이다.

대화 시나리오는 충남대학교 근처 작은 카페의 홍보 포스터를 제작하는 상황으로 설정했다. 사용자는 처음에는 부족한 정보만 제공했고, 이후 가게 분위기, 타겟 고객, 포스터 사용처, 필수 포함 내용, 이벤트 정보 등을 단계적으로 추가했다.

중간에는 “감성적인 톤에서 실용적인 톤으로 변경”하는 조건 변경을 넣었고, 후반에는 인스타그램 계정과 네이버 플레이스 링크가 아직 확정되지 않았다는 추가 정보를 제공했다.

이를 통해 AI가 다음 항목을 잘 수행하는지 검증했다.

- 이전 조건을 10턴 이상 유지하는가
- 중간에 변경된 조건만 반영하고 나머지 조건은 유지하는가
- 확인되지 않은 정보를 임의로 생성하지 않는가
- 사용자의 수정 요청을 반영하는가
- 환각 가능성이 있는 표현을 제거하거나 확인 필요로 분리하는가

**사용자**

- 타겟: 대학생
- 콘셉트: 조용한 휴식 공간

- 가게 이름
- 운영 시간
- 이벤트 기간
- 대표 음료 이름

다만 AI가 “이미 이전 답변에서 해당 표현들은 모두 제거했다”고 답변했는데, 실제 이전 답변에는 해당 표현이 포함되어 있었다. 따라서 수정 요청을 완전히 반영하지 못한 문제 지점으로 기록했다.


**AI 결과 요약**

AI는 인스타그램 계정명과 네이버 플레이스 링크를 임의로 생성하지 않았다.

포스터 하단에 다음과 같이 표시했다.

- 인스타그램 `[확인 필요 : 계정명]`
- 네이버 플레이스 `[확인 필요 : 링크 또는 검색어]`

또한 기존 조건인 충남대 근처 카페, 대학생 타겟, 인스타그램/당근마켓 사용처, 방문 인증 쿠폰 이벤트를 유지했다.

```text
지금까지 조건을 모두 반영해서 최종 포스터 문구를 다시 정리해줘. 제목 후보 3개, 부제목 2개, 본문 문구, CTA 3개, 확인 필요 정보까지 포함해줘.
```

- 부제목 2개
- 인스타그램용 본문
- 당근마켓용 본문
- CTA 후보 3개
- 미확인 정보 목록

다만 “오래 있어도 괜찮아요”라는 표현이 다시 등장했다. 이는 사용자가 직접 제공한 정보가 아니므로, 최종 분석에서는 미확인 정보로 분류했다.

| 오래 있어도 괜찮음 | 운영 정책에 해당할 수 있으므로 확인 필요 |

## 3-5. 실행 결과 분석

### 잘 수행된 점

- 정보가 부족한 첫 입력에서는 바로 결과를 생성하지 않고 확인 질문을 먼저 했다.
- 사용자가 추가로 제공한 조건을 반영하여 포스터 문구를 수정했다.
- 대표 음료 이름이 확정되지 않았다는 조건을 반영하여 임의의 음료명을 만들지 않았다.
- 이벤트 기간이 정해지지 않았다는 조건을 유지하고, 기간을 임의로 생성하지 않았다.
- 인스타그램 계정명과 네이버 플레이스 링크를 임의로 생성하지 않고 `[확인 필요]`로 표시했다.
- 10턴 이상의 대화 흐름에서도 대부분의 조건을 유지했다.

### 문제점

- 7턴에서 “자리 넓고”, “직원이 먼저 말 걸지 않는다”, “생각보다 맛있다는 말 많이 들어요”처럼 사용자가 제공하지 않은 표현이 생성되었다.
- 8턴에서 해당 표현이 이전 답변에 있었음에도 AI가 “이미 제거했다”고 잘못 응답했다.
- 11턴에서 “오래 있어도 괜찮아요”라는 미확인 표현이 다시 등장했다.

### 개선 방향

- 출력 전 자체 검토에서 사용자가 제공한 사실과 미확인 사실을 더 엄격히 분리해야 한다.
- “편안하다”, “부담 없다”와 같은 추상적 표현은 사용할 수 있지만, “자리 넓다”, “오래 있어도 된다”, “직원이 먼저 말 걸지 않는다”처럼 운영 정책이나 공간 특성을 단정하는 표현은 피해야 한다.
- 수정 요청이 들어온 경우, 이전 답변에 실제로 문제가 있었는지 먼저 인정하고 수정본을 제공해야 한다.
- 최종 문구에서는 `[확인 필요]` 표시를 더 적극적으로 사용해야 한다.

## 3-7. 실행 로그 결론

Claude Sonnet은 10턴 이상의 대화에서 대부분의 조건을 유지하고, 사용자가 추가한 정보를 반영하는 능력을 보여주었다. 특히 대표 음료 이름, 이벤트 기간, SNS 계정명, 네이버 플레이스 링크처럼 확인되지 않은 정보를 임의로 생성하지 않고 `[확인 필요]`로 표시한 점은 긍정적이었다.

다만 중간에 사용자가 제공하지 않은 공간 특성이나 응대 방식이 일부 생성되었고, 수정 요청을 완전히 정확하게 인식하지 못한 부분도 있었다. 따라서 최종 시스템에서는 자체 검토 항목을 강화하고, “사용자가 제공하지 않은 정보는 단정하지 않는다”는 규칙을 더 엄격히 적용해야 한다.

종합적으로 Claude Sonnet은 본 과업에서 가장 적합한 모델이지만, 실제 서비스 적용 시에는 최종 출력 전 미확인 사실 검토 단계를 반드시 포함해야 한다.

# 4. 환각 정의 및 창의적 생성과의 구분

## 4-1. 본 과제에서의 환각 정의

본 과제에서 **환각(Hallucination)**은 AI가 사용자가 제공하지 않았거나 검증되지 않은 정보를 실제 사실처럼 생성하는 현상으로 정의한다.

본 과제의 업무 과업은 **소상공인 홍보 포스터 문구 및 구성안 제작**이므로, 환각은 특히 실제 가게 정보와 관련된 사실 오류로 나타날 수 있다.

| 환각 유형 | 예시 | 문제점 |
|---|---|---|
| 미제공 가격 생성 | 사용자가 가격을 제공하지 않았는데 “아메리카노 2,500원”이라고 작성 | 확인되지 않은 가격 정보를 사실처럼 제시함 |
| 미확인 평판 생성 | “학생들이 가장 좋아하는 카페”라고 작성 | 실제 조사나 근거가 없음 |
| 순위/통계 생성 | “충남대 앞 1등 카페”라고 작성 | 검증된 순위 데이터가 없음 |
| 운영 정보 생성 | 운영 시간을 제공하지 않았는데 “밤 10시까지 운영”이라고 작성 | 실제 운영 시간과 다를 수 있음 |
| 이벤트 세부 내용 생성 | 이벤트 기간이 없는데 “이번 주말 한정”이라고 작성 | 사용자가 제공하지 않은 기간을 임의로 만듦 |
| 공간 특성 단정 | “자리 넓고 오래 머물 수 있음”이라고 작성 | 실제 공간 정책이나 좌석 정보를 확인하지 않음 |
| 고객 반응 단정 | “후기가 좋은 카페”라고 작성 | 후기 데이터가 제공되지 않음 |

따라서 본 과제에서 환각은 단순히 문장이 어색한 것이 아니라, **검증되지 않은 정보를 사실처럼 단정하는 것**으로 본다.

---

## 4-2. 창의적 생성과 환각의 차이

홍보 포스터 문구 제작은 창의적인 표현이 필요한 과업이다.  
그러나 창의적 문구 생성과 환각은 구분해야 한다.

| 구분 | 설명 | 예시 | 판단 |
|---|---|---|---|
| 창의적 생성 | 사용자가 제공한 사실을 바탕으로 표현을 다듬는 것 | “조용히 쉬어가기 좋은 카페” | 허용 |
| 창의적 생성 | 같은 의미를 더 자연스러운 홍보 문구로 바꾸는 것 | “공부하다 지친 날, 잠시 쉬어가세요” | 허용 |
| 환각 | 제공되지 않은 정보를 새 사실처럼 만드는 것 | “아메리카노 2,500원” | 금지 |
| 환각 | 검증되지 않은 평판이나 순위를 단정하는 것 | “충남대생이 가장 많이 찾는 카페” | 금지 |
| 환각 | 운영 정책을 임의로 만드는 것 | “오래 머물러도 괜찮은 공간” | 확인 필요 |

예를 들어 사용자가 “따뜻하고 조용한 분위기”라고 입력했다면, AI가 이를 “조용히 쉬어가기 좋은 공간”이라고 표현하는 것은 창의적 생성이다.  
하지만 사용자가 제공하지 않은 가격, 운영 시간, 평점, 순위, 이벤트 기간, 대표 메뉴명을 만들어내는 것은 환각이다.

---

## 4-3. 왜 환각을 사실 오류 중심으로 정의했는가

소상공인 홍보 포스터는 실제 고객에게 공개되는 홍보물이다.  
따라서 AI가 만든 문구가 자연스러운지뿐만 아니라, 실제 가게 정보와 일치하는지가 중요하다.

예를 들어 “따뜻한 분위기의 카페”는 사용자가 제공한 분위기를 바탕으로 한 표현이므로 허용 가능하다.  
하지만 “평점 4.9”, “충남대 앞 1등”, “이번 주말 20% 할인”처럼 실제 근거가 필요한 정보는 잘못 사용될 경우 소비자 오해, 허위 광고, 가게 신뢰도 하락으로 이어질 수 있다.

따라서 본 과제에서는 환각을 **창의적 표현의 실패**가 아니라 **검증되지 않은 사실 정보의 생성**으로 정의했다.

---

## 4-4. 모델 선정 이유와 환각 방지의 연결

최종 모델로 Claude Sonnet을 선정한 이유는 단순히 문구가 자연스러웠기 때문만은 아니다.  
본 과업에서는 실제 홍보물에 사용할 수 있는 문구를 생성해야 하므로 다음 능력이 중요했다.

1. 사용자가 제공한 정보와 제공하지 않은 정보를 구분하는 능력
2. 부족한 정보를 “확인 필요 정보”로 분리하는 능력
3. 과장 표현과 미확인 사실을 피하는 능력
4. 최종 출력 전 자체 검토를 수행하는 능력

Gemini 3 Flash와 GPT-5.4도 기본적인 문구 생성 능력은 충분했지만, Claude Sonnet은 확인 필요 정보와 자체 검토 항목을 가장 체계적으로 정리했다.  
특히 가격, 운영 시간, 이벤트 기간, 대표 메뉴명처럼 사용자가 제공하지 않은 정보를 임의로 만들지 않고 확인 필요 정보로 분리하는 점에서 본 과업의 환각 방지 목표와 가장 잘 맞았다.

따라서 Claude Sonnet 선정은 단순한 문구 품질 평가가 아니라, **소상공인 홍보물 제작에서 중요한 사실 정확성, 환각 방지, 실무 활용 가능성**을 함께 고려한 결과이다.

# 5. 심층 인터뷰 답변

## 5-1. 무료 모델만 사용할 수 있다면 품질을 어떻게 유지할 것인가

무료 모델만 사용할 수 있는 상황에서는 모델 자체의 성능보다 **프롬프트 구조와 검수 절차**를 강화하여 품질을 유지한다.

### 전략 1. 입력 템플릿을 더 구체화한다

무료 모델은 유료 모델보다 긴 문맥 유지나 세부 지시 준수 능력이 약할 수 있다.  
따라서 사용자가 입력해야 하는 정보를 구조화하여 모델이 임의로 추측할 여지를 줄인다.

```text
[가게 종류]
[홍보 목적]
[타겟 고객]
[포스터 사용처]
[원하는 분위기]
[반드시 포함할 정보]
[아직 확정되지 않은 정보]
[금지할 표현]
```

특히 “아직 확정되지 않은 정보” 항목을 별도로 두면, 모델이 가격, 운영 시간, 이벤트 기간 등을 임의로 생성할 가능성을 줄일 수 있다.

### 전략 2. 출력 형식을 고정한다

무료 모델은 자유 형식으로 요청할수록 결과가 흔들릴 수 있다.  
따라서 다음과 같이 출력 형식을 고정한다.

```text
1. 입력 정보 점검
2. 사용 가능한 정보
3. 확인 필요 정보
4. 포스터 문구 초안
5. 사용하면 안 되는 표현
6. 자체 검토
```

이렇게 하면 모델이 문구를 생성하기 전에 입력 정보를 먼저 점검하도록 유도할 수 있다.

### 전략 3. 한 번에 완성하지 않고 단계별로 생성한다

무료 모델에서는 한 번의 프롬프트로 최종 결과를 만들기보다 다음 단계로 나누는 것이 안정적이다.

```text
1단계: 입력 정보 점검
2단계: 핵심 메시지 생성
3단계: 제목/부제목 후보 생성
4단계: 본문과 CTA 생성
5단계: 환각 검수
```

단계별로 생성하면 오류가 발생한 지점을 더 쉽게 찾고 수정할 수 있다.

### 전략 4. 최종 검수 프롬프트를 별도로 사용한다

생성 프롬프트와 검수 프롬프트를 분리한다.

```text
아래 포스터 문구에서 사용자가 제공하지 않은 사실 정보가 있는지 찾아줘.
가격, 운영 시간, 평점, 순위, 이벤트 기간, 대표 메뉴명, 고객 반응, 공간 특성 중 임의로 생성된 내용이 있으면 표시하고 수정안을 제시해줘.
```

무료 모델을 사용할 때도 이러한 검수 단계를 추가하면 최종 품질을 높일 수 있다.

## 5-2. 업무 정책이 변경된다면 환각 감소 운영 전략을 어떻게 바꿀 것인가

홍보 포스터 제작 업무에서는 가게 정책이나 이벤트 조건이 자주 바뀔 수 있다.  
예를 들어 이벤트 기간이 변경되거나, 대표 메뉴가 바뀌거나, 운영 시간이 수정될 수 있다.

이런 정책 변경 상황에서는 다음과 같은 운영 전략을 사용한다.

### 전략 1. 변경 가능한 정보를 별도 정책 블록으로 관리한다

프롬프트 안에 고정 정보와 변경 가능 정보를 분리한다.

```text
[고정 정보]
- 가게 종류
- 브랜드 톤
- 타겟 고객
- 금지 표현

[변경 가능 정보]
- 운영 시간
- 이벤트 기간
- 대표 메뉴
- 할인 조건
- SNS 계정
```

이렇게 분리하면 변경된 정보만 업데이트하고, 모델이 오래된 정보를 계속 사용하는 문제를 줄일 수 있다.

### 전략 2. 최신 정책 확인 단계를 출력 전에 추가한다

모델이 포스터 문구를 작성하기 전에 다음 질문을 먼저 확인하도록 한다.

```text
포스터에 사용할 운영 시간, 이벤트 기간, 할인 조건, 대표 메뉴명이 최신 정보인지 확인했는가?
확인되지 않은 항목은 문구에 확정적으로 넣지 말고 [확인 필요]로 표시하라.
```

### 전략 3. 정책 변경 시 이전 출력물을 재검수한다

업무 정책이 바뀌면 기존에 만든 포스터 문구도 다시 검토해야 한다.

예를 들어 이벤트 기간이 바뀌었다면 다음 검수 프롬프트를 사용한다.

```text
아래 포스터 문구에서 변경된 이벤트 기간과 충돌하는 표현이 있는지 찾아줘.
기존 기간, 할인 조건, 대표 메뉴, 운영 시간과 관련된 문장을 모두 점검하고 수정안을 제시해줘.
```

### 전략 4. 확정 정보와 미확정 정보를 시각적으로 구분한다

최종 출력에서 확인되지 않은 정보는 다음처럼 명확히 표시한다.

```text
대표 음료: [확인 필요]
이벤트 기간: [확인 필요]
운영 시간: [확인 필요]
```

이를 통해 사용자가 실제 포스터 제작 전에 반드시 확인해야 할 항목을 놓치지 않도록 한다.

## 5-3. 문맥이 끊기는 문제를 해결하기 위해 프롬프트 구조를 어떻게 개선할 것인가

10턴 이상의 대화에서는 모델이 이전 조건을 잊거나, 중간에 바뀐 조건을 제대로 반영하지 못할 수 있다.  
이를 해결하기 위해 대화형 프롬프트 구조를 개선한다.

### 전략 1. 누적 조건 요약 블록을 사용한다

대화가 길어질수록 매 요청마다 현재까지 확정된 조건을 요약해서 제공한다.

```text
[현재까지 확정된 조건]
- 가게 종류: 충남대 근처 작은 카페
- 타겟 고객: 충남대 근처 대학생
- 포스터 사용처: 인스타그램, 당근마켓
- 분위기: 따뜻하지만 과장되지 않음
- 대표 음료명: 아직 미확정
- 이벤트: 인스타그램 방문 인증 사진 업로드 시 쿠폰 제공
- 이벤트 기간: 아직 미확정
- 금지 표현: 최고, 대박, 인생 카페, 1등, 가장 유명한
```

이 요약 블록을 반복해서 제공하면 모델이 이전 조건을 잊을 가능성이 줄어든다.

### 전략 2. 변경 사항만 따로 표시한다

중간에 조건이 바뀌는 경우, 전체 조건과 변경 조건을 분리한다.

```text
[변경된 조건]
- 기존 톤: 감성적
- 변경 후 톤: 실용적이고 부담 없는 느낌

[유지할 조건]
- 과장 표현 금지
- 미확인 가격/운영 시간/이벤트 기간 생성 금지
- 대학생 타겟 유지
```

이렇게 하면 모델이 변경된 조건만 반영하고, 나머지 조건은 유지하도록 유도할 수 있다.

### 전략 3. 출력 전 체크리스트를 포함한다

최종 답변을 생성하기 전에 다음 체크리스트를 내부적으로 확인하도록 한다.

```text
[출력 전 체크리스트]
1. 사용자가 제공하지 않은 가격을 만들지 않았는가?
2. 운영 시간을 임의로 만들지 않았는가?
3. 이벤트 기간을 임의로 만들지 않았는가?
4. 대표 음료명을 임의로 만들지 않았는가?
5. 금지 표현을 사용하지 않았는가?
6. 변경된 톤을 반영했는가?
7. 이전에 확정된 조건을 유지했는가?
```

### 전략 4. 최종 출력에 “반영한 조건”을 함께 표시한다

모델의 최종 답변에 반영 조건을 짧게 포함하면, 사용자가 문맥 유지 여부를 쉽게 확인할 수 있다.

```text
[반영한 조건]
- 대학생 타겟
- 따뜻하지만 과장되지 않은 톤
- 대표 음료명 미확정 처리
- 이벤트 기간 미확정 처리
- 방문 인증 쿠폰 이벤트 반영
```

이 방식은 긴 대화에서 모델이 어떤 조건을 실제로 반영했는지 확인할 수 있게 해준다.

## 5-4. 심층 인터뷰 종합 답변

본 시스템은 무료 모델만 사용하더라도 입력 템플릿, 출력 형식, 검수 프롬프트를 강화하여 품질을 유지할 수 있다.  
또한 업무 정책이 변경될 경우 고정 정보와 변경 가능 정보를 분리하고, 최신 정책 확인 단계를 추가하여 오래된 정보가 포스터 문구에 남는 문제를 줄일 수 있다.

긴 대화에서 문맥이 끊기는 문제는 누적 조건 요약 블록, 변경 사항 표시, 출력 전 체크리스트를 통해 완화할 수 있다.  
결국 본 과업에서 중요한 것은 모델이 문구를 얼마나 창의적으로 쓰는가보다, **사용자가 제공한 사실과 제공하지 않은 사실을 구분하고, 확인되지 않은 정보는 확정적으로 출력하지 않는 구조를 만드는 것**이다.

Mission01에서는 “소상공인 홍보 포스터 문구 및 구성안 제작 자동화”라는 과업을 선정하고, Gemini 3 Flash, Claude Sonnet, GPT-5.4 세 모델을 비교했다.

이후 Claude Sonnet을 기준으로 시스템 프롬프트를 설계하고, v1과 v2 프롬프트를 비교하여 입력 정보 점검, 확인 필요 정보 분리, 자체 검토 항목을 추가했다.

마지막으로 10턴 이상의 실행 로그를 통해 조건 유지, 조건 변경 반영, 미확인 정보 처리 여부를 검증했다. 실행 중 일부 환각 가능성이 있는 표현이 발생했지만, 이를 분석하고 개선 방향을 도출했다.

추가 보강 섹션에서는 본 과제에서의 환각을 “검증되지 않은 사실 정보를 실제 정보처럼 생성하는 것”으로 정의하고, 창의적 표현과 환각을 구분했다. 또한 무료 모델 사용 시 품질 유지 전략, 업무 정책 변경 시 환각 감소 운영 전략, 문맥 끊김 해결을 위한 프롬프트 구조 개선 방안을 제시했다.

따라서 최종 시스템은 소상공인이 홍보 포스터 문구와 구성안을 만들 때, 확인된 정보만 사용하고 미확인 정보는 별도로 표시하는 안전한 프롬프트 기반 도구로 활용할 수 있다.
